##### 스키마 Schema
ㄴ데이터베이스의 구조(개체, 속성, 관계)에 대한 정의<br/>
ㄴ메타 데이터 <br/>
ㄴ데이터 사전 Data Dictionary에 저장된다<br/>
ㄴ3단 구조:<br/>
외부 스키마 External schema - 공용의 의미보다는 어느 개인이나 특정 응용에 한정된 논리적 데이터 구조 <br/>
개념 스키마 Conceptual schema - 모든 응용프로그램이나 사용자들이 필요로 하는 데이터를 통합한 조직 전체의 데이터베이스 구조(데이터 개체, 관계,제약조건, 접근권한, 무결성규칙 등)를 정의한다 <br/>
내부 스키마 Internal schema - 물리적 저장 장치의 입장에서 본 데이터베이스 구조로 실제로 데이터베이스에 저장될 레코드의 형식을 정의하고 저장 데이터 항목의 표현 방법, 내부 레코드의 물리적 순서 등을 나타낸다<br/>

##### 시스템 카탈로그 
ㄴ시스템 자신이 필요로하는 스키마 및 여러가지 객체에 관련된 정보를 포함하는 시스템 데이터베이스이다<br/>
ㄴ데이터 사전<br/>
ㄴ시스템 카탈로그에 저장되는 내용을 메타 데이터라고 한다<br/>
ㄴ**무결성 유지**를 위하여 DBMS가 스스로 생성 관리 한다<br/>

##### 데이터베이스 설계:개>논>물
ㄴ**개념적 설계**<br/>
요구분석단계에서 나온 명세를 E-R 다이어그램과 같은 기법으로 기술하는 과정<br/>
요구 조건 분석 결과로 식별된 응용을 검토해서 이를 구현할 수 있는 트랜잭션을 명세로 기술<br/>
ㄴ**논리적 설계**<br/>
목표 DBMS에 종속적인 논리적 스키마 작성<br/>
논리적 데이터 모델로 변환<br/>
트랜잭션 인터페이스 설계(정규화)<br/>
스키마의 평가 및 정제<br/>
ㄴ**물리적 설계**<br/>
목표 DBMS에 종속적인 물리적 구조 설계<br/>
저장 레코드 양식 설계 및 레코드 집중의 분석/설계<br/>
파일 조직 방법과 저장 방법 그리고 파일 접근 방법 등 설정<br/>
응답 시간 효율화를 위한 접근 경로 설계<br/>
트랜잭션 세부 설계<br/>
트랜잭션 모델링(X) > 개념적 설계<br/>

##### 키
* 기본키 primary key : null값 안됨<br/>
ㄴ테이블 각 레코드를 고유하게 식별하는 필드나 필드의 집합<br/>
ㄴ테이블에 기본키 설정은 **필수가 아니다**<br/>
ㄴ기본키 설정하지 않고도 다른 테이블과의 관계 설정 가능<br/>
ㄴ관계가 설정되어 있는 테이블에서 기본키설정 해제시 관계는 유지<br/>
ㄴ데이터가 이미 입력된 필드도 기본키로 지정 가능, 기본키는 변경될 수 있다<br/>
* 외래키 Foreign key : null값 가능<br/>
ㄴ과계형 데이터 모델에서 한 릴레이션의 외래키는 참조되는 릴레이션의 기본키와 대응되어 릴레이션 간에 참조 관계를 표현하는데 사용된다<br/>
ㄴ외래키를 포함하는 릴레이션이 참조하는 릴레이션이 되고, **대응되는 기본키를 포함하는 릴레이션**이 **참조 릴레이션**이 된다<br/>
* 후보키<br/>
ㄴ기본키가 될 수 있는 속성 =기본키+대체키<br/>
ㄴ릴레이션에 있는 모든 튜플에 대해 유일성과 최소성을 모두 만족시켜야 한다<br/>
</>슈퍼키: 유일성(O), 최소성(X)<br/>

##### 무결성의 종류
ㄴ개체 무결성: 기본키의 값은 null값이나 중복값을 가질 수 없다는 제약 조건<br/>
ㄴ참조 무결성: 참조할 수 없는 외래키 값을 가질 수 없다는 제약 조건<br/>
ㄴ도메인 무결성: 각 속성값은 해당 속성 도메인(값들의 영역)에 지정된 값이어야 한다는 제약 조건<br/>
   ex) 1~4학년 </> 8학년(x)<br/>

##### 정규화 Normalization  -> 결과물: 정규형
<img width="411" height="451" alt="image" src="https://github.com/user-attachments/assets/07dca0a2-a9ad-44d9-987e-36f7abbb7056" /><br/>
ㄴ 데이터베이스 논리적 설계단계에서 실행 <br/>
ㄴ 이상현상을 해결하기 위해 스키마를 더 작은 속성을 세트로 쪼개는 것 (결정자 > 종속자: 종속관계가 잘못 설정된 경우)<br/>
ㄴ 데이터 구조의 안정성 최대화<br/>
ㄴ 중복을 배제하여 삽입/삭제/갱신 이상을 방지 <br/>
ㄴ 데이터 삽입 시 릴레이션을 재구성할 필요성을 줄인다<br/>
ㄴ1NF = 도메인이 원자값(문제점: 부분함수 종속)<br/>
ㄴ2NF = 1NF + 완전함수 종속(문제점: 이행함수 종속)<br/>
ㄴ3NF = 2NF + 이행적 함수 종속 해결(문제점: 모든 결정자 <>후보키)<br/>
ㄴBCNF = 3NF + 모든 결정자=후보키로 해결 <br/>
ㄴ4NF = BCNF + 다치종속제거<br/>
ㄴ5NF = 4NF + 조인 종속성 제거 <br/>

##### 병행제어/로킹
* 병행 제어의 목적<br/>

ㄴ데이터베이스 일관성 최대화(무결성 유지)<br/>

##### 분산 데이터베이스의 투명성 
ㄴ위치 투명성 Location Transparency : 하드웨어와 소프트웨어의 물리적 위치를 사용자에게 노출할 필요x<br/>
ㄴ중복(복제) 투명성 Replication Transparency : 사용자에게 통지할 필요 없이 시스템 안에서 파일/자원 부가적인 복사 자유롭게 수행<br/>
ㄴ병행 투명성 Concurrency Transparency: 다중 사용자들이 자원을 자동으로 공유<br/>
ㄴ장애 투명성 Failure Transparency: 사용자들이 시스템 장애 발생 부분에 대해 알 필요 x<br/>
ㄴ분산 데이터베이스의 구성 요소 : 분산 처리기/분산 데이터베이스/통신 네트워크/분산 트랜잭션<br/>
ㄴ분산 데이터베이스의 구조: 전역/분할(단편화)/할당/지역 스키마<br/>
